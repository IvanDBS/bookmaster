Аудит проекта BookMaster и дорожная карта

1) Цели аудита
- Выявить критические уязвимости безопасности и целостности данных
- Обозначить архитектурные слабые места и технический долг
- Сформировать приоритизированный план улучшений для выхода на рынок против alteg.io

2) Критичные уязвимости и риски (исправить в первую очередь)
- Безопасность аутентификации: регистрация не гарантированно выдает JWT токен; токен хранится в localStorage → риск XSS/угон сессии
- Отсутствует rate limiting/блокировка брутфорса (нет Rack::Attack, Devise lockable)
- Публичная выдача мастеров раскрывает PII (телефон, адрес) без аутентификации; нет пагинации/ограничения выдачи
- CORS и BASE_URL захардкожены для dev, credentials: true при использовании Bearer токенов → потенциальные риски в проде
- Синхронизация слотов: массовый сброс is_available для всех слотов дня стирает lunch/blocked статусы; уязвимость целостности расписания
- Несогласованность длительности брони: Booking.end_time всегда +60 минут, игнорируется Service.duration → неверные окна и конфликт с цепочными слотами
- Конкурентность при создании брони длиннее 1 слота: цепочка слотов рассчитывается до транзакции; существует окно гонки
- Seeds опасны: полное удаление данных без ENV-гейта (недопустимо для staging/prod)
- Логи: избыточные подробности и потенциальное логирование PII в проде; нет маскировки

3) Быстрые правки (1–2 дня, максимальный ROI)
- Auth/JWT: при регистрации вызывать sign_in и гарантированно возвращать JWT; выровнять формат ответа с login
- Ограничить публичные поля мастеров (только минимально необходимые), добавить пагинацию и базовые лимиты скорости
- Слоты: при reconcile не трогать слоты типа lunch/blocked; обновлять только work; не сбрасывать is_available для не-рабочих слотов
- Длительность брони: вычислять end_time на основе service.duration; при создании брони проверять и атомарно резервировать всю цепочку слотов внутри транзакции
- CORS/конфиг: вынести origins/BASE_URL в ENV; в проде выключить credentials при Bearer; включить force_ssl; CSP заголовки
- Rate limiting и защита входа: Rack::Attack (лимит по IP/логину), Devise lockable
- Хранение токена: перейти на httpOnly secure cookie (минимум — добавить строгий CSP и очистку localStorage при 401)
- Логи: включить Lograge, маскировать PII, понизить уровень подробности в проде; подключить Sentry
- Seeds: добавить защиту (RAILS_ENV=development guard), seed-idempotency, флажок CONFIRM_RESET
- Разбить MasterDashboard.vue на подкомпоненты (календарь, слоты, модалки, фильтры, список услуг); подключить Refactored версию в роутере

4) Улучшения на 2–4 недели
- Единая авторизация через Pundit-политики для всех ресурсов; убрать дубли ручных проверок
- Перенос бизнес-логики слотов/бронирований в сервис-объекты; покрыть юнит-тестами гонки и границы
- Унификация времени: ISO 8601, явная тайм-зона, одинаковые форматы везде (включая public slots)
- OpenAPI/Swagger спецификация + автоген клиента; статическая валидация контрактов
- Централизованный API-клиент на фронте (интерсепторы 401/429, backoff, единая обработка ошибок)
- E2E (Cypress/Playwright) для ключевых пользовательских сценариев; Storybook для UI-компонентов
- Нотификации: интеграция email/SMS/Telegram для подтверждений и напоминаний (через фоновые задачи)
- Фоновые задачи (Sidekiq) для синхронизаций/рассылок; health/ready пробы; Docker-образ и CI/CD
- Многоязычность (i18n) фронта и бэка; подготовка к брендированным витринам

5) Дорожная карта фич для конкурентоспособности с alteg.io
- Календарь и расписания: 
  - Длительности услуг, буферы между услугами, повторяющиеся шаблоны смен, праздничные дни
  - Ресурсы (кабинеты/оборудование), запрет овербукинга, групповые брони
  - Двусторонняя синхронизация с Google/Apple Calendar, ICS-ссылки
- Платежи: 
  - Предоплата/депозиты, штраф за no-show, промокоды/пакеты/абонементы
  - Интеграции с PSP, счета/чеки (региональные требования)
- CRM и коммуникации: 
  - Карточка клиента, история, теги, заметки, сегментация
  - Шаблоны сообщений, напоминания (email/SMS/мессенджеры), отзывы/рейтинг
- Команды и мульти-локации: 
  - Роли (владелец/админ/мастер/ресепшн), права доступа
  - Несколько филиалов, общая аналитика, распределение бронирований
- Аналитика: 
  - Выручка, загрузка, no-show rate, LTV, удержание, эффективность акций; экспорт
- Продукт/UX: 
  - PWA (офлайн-доступ, пуш-уведомления), мобильные приложения
  - Публичные профили мастеров, быстрые ссылки/QR для записи

6) Техническая дорожная карта — Backend
- Валидация/сертификация API: OpenAPI + контракты, единый формат ошибок
- Auth: httpOnly secure cookies для JWT или короткоживущие JWT + refresh; lockable; Rack::Attack
- Политики доступа: Pundit для Users/Services/Bookings/TimeSlots/WorkingSchedules/Exceptions
- Бизнес-логика: 
  - Сервис для создания брони с атомарной резервной цепочкой слотов 
  - Сервис для reconcile, уважающий типы слотов и длительности
- Форматы времени: строго ISO 8601, тайм-зона приложения, явные поля date/time где нужно
- Нотификации: фоновая очередь (Sidekiq) + адаптеры email/SMS/Telegram
- Наблюдаемость: Sentry, Lograge, структурированные логи, метрики (Prometheus)
- Миграции/данные: безопасные seeds, миграции для дополнительных полей (price на Booking при необходимости, refresh tokens и т.п.)

7) Техническая дорожная карта — Frontend
- Разборка MasterDashboard.vue на модули: Calendar, DaySlots, BookingActions, Exceptions, Services
- Централизованный API-клиент: интерсепторы 401/429, ретраи, троттлинг
- Хранение сессии: отказаться от localStorage (использовать cookie); как минимум — строгий CSP, XSS guard, sanitization
- Состояние: Pinia-модули по доменам (auth, calendar, bookings, services)
- UI/UX: загрузчики, ошибки, пустые состояния; i18n; доступность (a11y)
- Тесты: компонентные + e2e; Storybook для сложных компонентов

8) DevOps/Инфраструктура
- CI: линтеры, unit, e2e, security audit (Brakeman, Bundler Audit, npm audit)
- CD: сборка Docker-образа, миграции, миграция секретов (Rails credentials), ENV для CORS/BASE_URL
- Мониторинг/алертинг: Sentry, uptime, health/readiness/liveness пробы
- Бэкапы БД и политика восстановления; миграции под нагрузкой

9) Тестирование и качество
- Unit тесты: сервисы бронирований/слотов, политики Pundit, контроллеры
- Контрактные тесты API по OpenAPI
- E2E сценарии: авторизация, просмотр календаря, создание/подтверждение/отмена брони, управление слотом
- Нагрузочное тестирование: список мастеров/слотов, создание брони в гонке

10) KPI/Метрики
- Технические: 95% успешных бронирований; <1% ошибок 5xx; p95 латентность API < 300ms
- Продуктовые: конверсия записи > 5%; no-show < 8% с напоминаниями; удержание D30 > 30%

11) План релизов (milestones)
- M1 (1–2 недели): быстрые фиксы безопасности/целостности, разборка Dashboard, Lograge+Sentry, Rack::Attack, Pundit-скелет, OpenAPI скелет
- M2 (3–5 недель): сервисы бронирований/слотов, httpOnly cookie, централизованный фронт-клиент, e2e, нотификации (email), i18n
- M3 (6–10 недель): платежи/депозиты, календарь-синк Google, роли/мульти-локации, аналитика, PWA

12) Риски и меры
- Конкурентные гонки при бронированиях → транзакции, локи, идемпотентность
- Утечка PII → минимум данных публично, маскировка логов, DPA/GDPR
- Срыв сроков из-за рефакторинга → итеративная разбивка, фичефлаги

- Изменить reconcile слотов: не трогать lunch/blocked; считать по service.duration
- Вынести CORS/BASE_URL в ENV; включить force_ssl, миним. CSP
- Включить Rack::Attack и Devise lockable
- Ограничить публичные поля мастеров + пагинация
- Подключить Sentry и Lograge в прод-конфиге
- Начать разборку MasterDashboard.vue на модули и переключить роутер на рефакторенную версию
